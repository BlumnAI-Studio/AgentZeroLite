# v1.7.1 — Unit-test policy relaxed: Author-verify exception

**Released**: 2026-05-23
**Bump rationale**: Patch — refines the existing unit-test execution
policy with one narrow auto-fire exception. No new agent, no new
engine, no removed trigger. Default behavior (no auto-run) is unchanged
for the vast majority of code paths.

## Why this change

The harness has run on "tests are explicit-only" since v1.3.0. The
default holds well — on-device LLM smoke tests (Whisper / Gemma /
Webnori) are slow enough that reflexive `dotnet test` runs burn the
user's CPU/RAM with low marginal signal.

But the policy had a blind spot: when Claude *authored a brand-new
test* mid-task, the test could land green in CI because nobody ever
actually ran it. False-pass tests are worse than no test — they look
like coverage while asserting nothing.

The fix is narrow: when Claude writes a new test in the same task,
filter-run just that one test (or one filter expression covering the
batch added in the task) before reporting done. One run, scoped tight,
audited in the log. LLM-smoke classes still ask the user first because
the original CPU concern is unchanged for them.

## What's new

### `harness/knowledge/test-runner/unit-test-policy.md` (refined)

- Added the **Author-verify exception** clause to the Rule section
  with explicit bounds: scope = ONLY the new class/method, one
  filtered run per task, LLM-smoke requires user permission, pure
  refactors do not fire.
- Updated the "What still runs automatically" table:
  - New row: `dotnet test` filtered to a newly-authored non-LLM test
    → **yes** (Author-verify).
  - New row: `dotnet test` filtered to a newly-authored LLM-smoke
    test → **ask first**.
- Added boundary-case examples for "Claude wrote a new `[Fact]`",
  "Claude added a new test class", and "Claude renamed an existing
  test" so the discriminator is concrete.

### `harness/agents/test-runner.md` (new mode)

- Added `auto_invoke_on:` frontmatter declaring the Author-verify
  auto-fire condition + LLM-smoke ask-first guard.
- Description updated: "Default = never auto-runs. One narrow
  exception (Author-verify)…"
- Triggers table grew from 3 modes to 4 — added **Author-verify**
  row.
- New Procedure section "Author-verify (auto, narrow)" detailing:
  - When the mode fires (new `[Fact]`/`[Theory]`/test class in
    the task diff; renames/moves don't count).
  - LLM-smoke guard wording (the verbatim question to ask).
  - Filter construction (`~ClassName.MethodName` for single,
    `~ClassA|~ClassB` for batches).
  - Scope discipline (at most once per task, never widens).
- "What the Runner does NOT do" updated with three new clauses:
  no auto LLM-smoke even under Author-verify, no scope widening,
  refactors of non-test files still don't trigger.
- Log format `mode:` enum extended to include `author-verify`.
- Evaluation rubric: added "Author-verify narrowness" and
  "LLM-smoke guard" axes (both Pass/Fail).

## What did NOT change

- `pre-commit-review` (code-coach) still does not invoke tests.
- `release-build-pipeline` still has no test phase.
- `test-sentinel` still does not execute — Author-verify is owned
  by `test-runner`.
- `build-doctor` still runs `dotnet build` only.
- All existing user-initiated triggers (Scoped / Full / History)
  unchanged.

## Migration impact

Zero. Any existing log under `harness/logs/test-runner/` is still
valid — the `mode:` enum was widened, not narrowed. New logs from
2026-05-23 onward may carry `mode: author-verify`.
