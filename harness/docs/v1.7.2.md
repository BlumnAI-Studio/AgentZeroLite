# v1.7.2 — codescan integration + M0023 architecture visualization

**Released**: 2026-05-25
**Bump rationale**: Patch — adds a new code-analysis primitive
(`codescan` CLI) to the documentation workflow and lands the first
mission that exercises it end-to-end. No new agent, no new engine,
no removed trigger. Existing missions, build pipeline, and review
flows are untouched.

## Why this change

The harness so far covered build, review, test, and security paths
but had no canonical way to ask "what does this codebase actually
contain right now?" Operators were either eyeballing `find`/`rg`
output or guessing at class relationships. M0023 introduces
[`codescan`](https://github.com/psmon/CodeScan) as the standard
read-only structural lens — it scans the repo, indexes files /
classes / methods / inheritance edges into SQLite, and answers
Cypher-subset graph queries.

Pairing codescan with Pencil closes the loop: structural facts come
out of the index, visual maps go into `Docs/design/M{NNNN}-*.pen`,
the harness-view Missions card pairs the design with the mission
record automatically (Rule 5).

### codescan as a harness-side primitive

- `codescan project-update <id> --source` refreshes the index of
  any tracked repo (git pull + full scan). Project #5 = AgentZero
  Lite (596 files / 3,478 methods at 2026-05-25).
- `codescan query "MATCH ..."` exposes a Cypher-subset over nodes
  (`project`, `directory`, `file`, `class`, `method`, `type`,
  `module`, …) and edges (`contains`, `defines`, `imports`,
  `inherits_or_implements`, `uses_type`, …).
- Pattern caveat surfaced during M0023: **AND-only `WHERE`**;
  `OR` is rejected with `Unsupported WHERE condition`. Splitting
  into two queries is the workaround.

### Mission-protocol contract — backward-compat note

M0023's source file landed without YAML frontmatter (operator wrote
a plain brief). The mission-dispatch engine now treats that as a
recoverable case: tamer inserts the missing frontmatter at status
flip time rather than rejecting. This mirrors the existing rule
that mission filenames may be Korean while their slugs stay English
— body shape is informal, lifecycle is formal.

## What did NOT change

- `mission-dispatch` engine triggers, evaluation rubric, and
  step ordering — unchanged.
- `pre-commit-review`, `release-build-pipeline`, `crash-dump-triage`
  engines — unchanged.
- All existing agents (tamer / security-guard / build-doctor /
  test-sentinel / test-runner / code-coach) — unchanged.
- harness-view indexer regex (`^M\d+\b` for mission records,
  `M{NNNN}-` prefix for design pairing) — unchanged. M0023 conforms
  on both axes.

## Migration impact

Zero. Existing missions and logs continue to work. Operators that
want the new lens just need `codescan` on PATH and one
`codescan project-update <id> --source` per repo to seed the index.
